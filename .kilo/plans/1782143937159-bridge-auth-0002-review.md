# Adversarial Plan Review: BRIDGE-AUTH-0002 Final Revision

## Verdict
**accept_with_repairs**

The plan is technically sound and closely follows the BRIDGE-CORE-0001 store-layer pattern. The cryptographic design (AES-256-GCM at rest, HMAC-SHA256 request verification, server-owned nonce TTL, no offline verifier) is correct. However, there are several schema inconsistencies, test gaps, and serialization risks that must be closed before implementation to prevent security defects or data leaks.

---

## Critical Blockers
None. The prior critical blockers (plain SHA256 offline verifier, client-derived expiry, plaintext secrets) are fully removed from this revision.

---

## High-Priority Repairs

1. **Schema gap: missing `CHECK(expires_at > timestamp)` in the proposed nonce table**
   The plan text calls `CHECK(expires_at > timestamp)` an "expected minor schema patch" but the proposed `CREATE TABLE bridge_auth_nonces` omits it. Either add the CHECK to the proposed schema, or explicitly state in the plan that it is intentionally omitted and justify why. Without it, the `timestamp` column is unconstrained, which undermines the nonce table invariant.

2. **JSON serialization leak risk for `BridgeAuthKey` sensitive fields**
   The plan requires `ListBridgeAuthKeys` to exclude secret material, but `GetBridgeAuthKey` returns the full struct containing `SecretKeyEncrypted` and `SecretKeyNompare` and the API handler must not serialize them. The plan should mandate either:
   - `json:"-"` struct tags on `SecretKeyEncrypted` and `SecretKeyNonce`, or
   - Explicit tests verifying that `GET` API responses do NOT contain `secret_key_encrypted`, `secret_key_nonce`, or the decrypted secret.

3. **No test for nil encryption service**
   Add `TestBridgeAuthMiddlewareNilEncryptionService` (or equivalent) to verify `enc == nil` returns HTTP 500 and `ErrBridgeAuthSecretUnavailable` without panicking.

4. **Inactive tenant rejection not specified**
   The middleware resolves `tenant_id` server-side, but the plan does not explicitly require rejecting tenants where `IsActive = false`. Add explicit wording and a test (`TestBridgeAuthRejectsInactiveTenant`).

5. **Timestamp column justification and parsing validation**
   The nonce table includes a `timestamp INTEGER NOT NULL` column. The plan must explicitly state this stores the client-provided `X-Bridge-Timestamp` header value. Additionally, the middleware must reject non-numeric timestamp headers before `strconv.ParseInt` to avoid panics or unexpected errors.

---

## Medium-Priority Repairs

1. **Explicit `FindBridgeAuthKey` filter fields**
   The plan should specify `FindBridgeAuthKey` includes at least `ID *int32`, `TenantID *int32`, `KeyID *string`, and `Status *string` so the middleware can efficiently query active keys.

2. **`last_used_at` update failure semantics**
   Explicitly state that if `UpdateBridgeAuthKeyLastUsed` fails after a valid signature and successful nonce insert, the middleware must return the success response and log the error rather than failing the client request.

3. **Audit logging requirements**
   The plan says "Do not log raw secret, decrypted secret, raw body, HMAC signature, nonce, or canonical string" but does not specify what *should* be logged. Require audit logs for: `key_id`, `tenant_slug`, route path, result (`accepted` / `rejected`), and rejection reason (without secret material).

4. **Consider removing the `timestamp` column from `bridge_auth_nonces`**
   Server-side replay protection only needs `tenant_id`, `key_id`, `nonce`, `created_at`, and `expires_at`. The client timestamp adds audit surface area but no enforcement value (the middleware already validates freshness). If retained, justify its presence.

5. **Content-type canonicalization edge case**
   Add a test ensuring `application/json; charset=utf-8` is canonicalized to `application/json` in the signed string, and that `application/json; charset=iso-8859-1` is rejected because the type is still `application/json` but the server enforces UTF-8. (RFC 8259 mandates UTF-8 for JSON.)

## Critical Blocker Re-Check

- **Plain `SHA256(raw_secret)` removed**: Plan explicitly says "Store no offline verifier secret_hash." Confirmed removed.
- **Client-derived expiry removed**: Plan says "never derive expires_at from client timestamp" and uses `server_now + 15 minutes`. Confirmed.
- **Nonce expiry server-owned**: Plan says `expires_at = server_now + 15 minutes`. Confirmed.
- **No raw secret returned after creation**: Plan says "raw secret returned exactly once." Confirmed.

## SQLite Schema Review

- `bridge_auth_keys` and `bridge_auth_nonces` are sufficient for the auth foundation.
- `secret_hash` correctly omitted.
- `secret_key_nonce` length CHECK (`= 12`) matches AES-GCM nonce size.
- Key/nonce length checks (`BETWEEN 16 AND 128`) are adequate.
- Revocation invariants (`active AND revoked_at IS NULL` vs `revoked AND revoked_at IS NOT NULL`) are correct.
- Composite FK `(tenant_id, key_id) REFERENCES bridge_auth_keys` is valid in SQLite.
- Indexes `idx_bridge_auth_keys_tenant_status`, `idx_bridge_auth_nonces_tenant_key`, and `idx_bridge_auth_nonces_expiry` are appropriate.
- Nullable `label` is acceptable.
- Migration path follows repo convention (`0.25/30__bridge_auth_foundation.sql`).
- `LATEST.sql` update required and tested.
- **Deficiency**: `CHECK(expires_at > timestamp)` missing from proposed schema despite being called out as expected.

- `EncryptionService` (AES-256-GCM + Argon2id) is the existing, approved pattern.
- `EncryptionMasterKey` + `SystemSecret.EncryptionSalt` is the established root-secret model in this codebase.
- Nil `enc` fail-safe is explicitly required.
- Raw secret returned exactly once on creation.
- `Get`/`List` exclude raw secret per plan, but `json:"-"` tag requirement is missing.
- No fingerprint, no `secret_hash`, no offline verifier.
- `secret_key_nonce` is stored and used for decryption; this is the same pattern as `TenantConfig.OpenRouterAPIKeyNonce`.

- `io.LimitReader(body, maxBodyBytes + 1)` before read — correct.

## HTTP Error and Logging Review

- Error mapping follows plan:
  - `400`: malformed request
  - `401`: auth failures (timestamp, signature, key not found, key revoked)
  - `409`: replay
  - `500`: secret unavailable
  - `501`: unsupported database
- Client responses must not expose raw internal error strings.
- Log sanitization required but log *content* requirements not specified (see Medium-Priority Repair #3).

## Revised Acceptance Criteria

1. **Schema**: `bridge_auth_keys` and `bridge_auth_nonces` tables created with all CHECK constraints, including `CHECK(expires_at > timestamp)` OR explicit documented justification for omission.
2. **Secret Safety**: `SecretKeyEncrypted` and `SecretKeyNonce` are tagged `json:"-"` in the store struct, OR all API handlers returning key metadata explicitly exclude them.
3. **Crypto**: Raw secret returned exactly once; never stored, logged, or returned by `Get`/`List`; AES-256-GCM nonce length enforced; no `secret_hash`.
4. **Middleware**: 
   - Rejects nil encryption service with 500.
   - Rejects inactive tenants.
   - Rejects non-numeric `X-Bridge-Timestamp`.
   - Validates timestamp within ±5 minutes.
   - Limits body to 1 MiB before read.
   - Canonical string uses `BCHAT-BRIDGE-V1`, exact route slug, upper method, `EscapedPath()`, canonical content type, unix timestamp, nonce, base64url SHA256 body hash, no trailing newline.
   - Uses `hmac.Equal`.
   - Inserts nonce only after valid signature.
   - Returns 409 on replay.
   - Restores request body.
5. **Store**: 
   - All methods present on `Driver` and `Store`.
   - PG/MySQL stubs return `ErrBridgeUnsupportedDatabase`.
   - Revocation idempotent.
   - `CleanupBridgeAuthNonces` deletes expired rows.
6. **Routes**: No new bridge operational endpoints registered. `ChatExternal` behavior unchanged.
7. **Tests**: All listed tests pass, including race tests. Coverage proves no offline verifier, server-time TTL, nonce not consumed on invalid signature, concurrent replay safety, route scope discipline, and no secret/signature/canonical-string logging.
8. **Migrations**: `30__bridge_auth_foundation.sql` applies cleanly and `LATEST.sql` is in sync.

---

## Final Recommendation
**Patch first.** The plan should be revised to add the missing CHECK constraint (or explicit omission justification), mandate `json:"-"` on sensitive store struct fields (or explicit handler sanitization), add the nil-encryption-service test, and add explicit inactive-tenant and non-numeric-timestamp handling. Once these repairs are incorporated, the plan is ready for implementation.
