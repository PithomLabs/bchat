# Test Failure Review: store/test

## Findings

### Failure 1 — `TestGetCurrentSchemaVersion`
- **Root cause:** Correct. The test hardcodes `0.24.2` but migration count now yields `0.25.29`.
- **Proposed fix:** Update expected value to `0.25.29`.
- **Nit:** Fragile — it will break again on the next migration. Better long-term pattern: derive expected dynamically from counting `store/migration/sqlite/0.25/*.sql`, or compare against `version.GetCurrentVersion("dev")`+patch-count. For now, hardcoded update is acceptable as an unblock.

### Failure 2 — `TestTicketForeignKeyConstraints`
- **Root cause:** Correct. `LATEST.sql` lacks FK constraints on `tickets(creator_id)` and `tickets(assignee_id)`; migration `04__tickets_add_foreign_keys.sql` adds them but fresh DBs load `LATEST.sql` directly.
- **Proposed fix:** Add the FK constraints to `LATEST.sql` and the missing `idx_tickets_assignee_id` index.
- **Nits:**
  1. **Behavioral note:** `ON DELETE CASCADE` on `creator_id` means deleting a user will cascade-delete their tickets. This is consistent with the migration intent but is a real behavior change for any future fresh-install or reset-DB path. Confirm this is the intended policy.
  2. **Inconsistency note:** `memo`, `resource`, `activity`, `webhook`, `reaction`, and `inbox` all have bare `INTEGER` `creator_id` columns without FK constraints in `LATEST.sql`. Adding FKs only to `tickets` makes the schema partially normalized. The test only cares about tickets, so this is fine as a scoped fix, but it's a known schema debt.
  3. **FK pragma confirmation:** `test.txt` shows `Foreign keys pragma: 1 (should be 1)` — SQLite FK enforcement is enabled in the test harness, so the constraint fix will actually be enforced.

### Ordering
Proposed order (LATEST.sql first, then test) is correct: LATEST.sql is the source of truth and the schema-version test merely asserts a derived value.

## Recommendation

**Approve with nits.** Action 1 needs a follow-up to remove fragility. Action 2 is the correct scope for this bug, but requires explicit sign-off that `ON DELETE CASCADE` on `creator_id` matches the intended data-retention policy. If there is any concern about cascade-delete, the safer alternative in Action 2 is still preferred, or the FK test can be adjusted to match the current schema.
